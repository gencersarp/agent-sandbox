You are a senior backend/security engineer building a CI integration.

PROJECT
Build “Agent Sandbox for GitHub Actions”: a framework + GitHub Action that runs LLM agents safely inside CI with strict capability manifests, resource limits, and explicit human approval of proposed changes.

HIGH‑LEVEL OBJECTIVE
Deliver an MVP that:
- Lets a repository define, via YAML, exactly what files and directories an AI agent can read and write, what commands it may run, and any network restrictions.
- Spins up an isolated container in a GitHub Action job, runs an agent process inside it with those constraints, and produces:
  - A set of proposed file patches and comments.
  - A summary report of what it tried to do.
- Never pushes changes directly to the default branch; output is staged as a branch or artifact for human review.

TARGET USER / UX CONSTRAINTS
- Target user: developers who want “AI PR bot” behavior in CI, but are worried about giving it broad repo access or shell capabilities.
- The integration must look like a normal GitHub Action in a workflow YAML.
- Developers should be able to get a simple use case (e.g., auto‑fix lint) running in under 15 minutes.
- Security posture must be conservative by default.

TECH STACK
- Runtime: Docker‑based container in GitHub Actions (ubuntu‑latest or similar).
- Agent Runner: Python or Node.js (choose one and stick to it).
- Policy Engine: Pure code + a YAML capability manifest; no complex DSL.
- LLM interface: Generic HTTP client; for MVP assume OpenAI‑compatible endpoint configured via secrets.
- Output format: 
  - Unified diff patches of modified files.
  - JSON summary report.

CORE FUNCTIONAL REQUIREMENTS

1) Capability manifest format
- Define a YAML schema (e.g., `.agent-sandbox.yml`) that supports:
  - Allowed_paths:
    - read_only: list of glob patterns.
    - read_write: list of glob patterns (where agent can modify/create files).
  - Allowed_commands: list of shell commands or regexes the agent may invoke (e.g., `npm test`, `pytest`, `ruff`, etc.).
  - Network:
    - allowed_domains: list of hosts the agent may reach (or empty for no network).
  - Agent_task:
    - short description of what the agent should do (e.g., “fix lint errors reported by ruff in this repo”).
    - optional additional instructions.
- Provide a small JSON schema or runtime validation for this file.

2) GitHub Action integration
- Provide:
  - A reusable composite or Docker Action in this repo, installable via `uses: org/agent-sandbox-action@v0`.
  - Example workflow:

      - name: Agent Sandbox Lint Fix
        uses: org/agent-sandbox-action@v0
        with:
          manifest_path: ".agent-sandbox.yml"

- The Action should:
  - Checkout the repo.
  - Load the manifest.
  - Start a sandbox container with a limited filesystem view (enforced via mount options and path checks in code).
  - Run the agent runner with:
    - The manifest.
    - A summary of the diff vs base branch (for context).
  - Capture any file changes inside the allowed write paths.
  - Produce:
    - Patch file as an artifact.
    - JSON report summarizing changes.

3) Sandbox enforcement
- Within the agent runner:
  - Wrap filesystem operations so that any attempt to read/write outside allowed paths fails fast.
  - If you allow shell commands:
    - Provide a tiny wrapper that only executes commands enumerated in `allowed_commands`.
    - Capture stdout/stderr for the report.
  - Network:
    - For the MVP, enforce by forbidding all network requests from the agent process unless an env var / config explicitly whitelists domains (e.g., by blocking HTTP client calls or checking URLs).
- Do not attempt to implement bulletproof OS‑level sandboxing; focus on a strong, explicit policy at the application layer with clear documentation of limitations.

4) Agent behavior
- The agent should:
  - Read the manifest, repository context (e.g., git diff summary, list of files), and task description.
  - Plan a minimal set of edits to satisfy the task (e.g., run linter, then fix issues in specific files).
  - Make file edits only through the sandboxed file API, generating diffs.
- For the MVP, support one or two opinionated workflows:
  - Auto‑fix linting issues (ruff/eslint).
  - Update CHANGELOG.md based on commit messages.

5) Output and reporting
- On completion:
  - Generate a unified diff of all changes in allowed write paths.
  - Write a JSON report with:
    - Files modified.
    - Commands executed and their exit codes.
    - Any errors encountered.
    - High‑level natural language summary (generated by the LLM).
- The Action should:
  - Upload the patch + report as artifacts.
  - Optionally print a concise summary in the job log with a link to artifacts.

6) Developer experience
- Provide:
  - Example repo with a working `.agent-sandbox.yml` for:
    - Lint autofix.
    - Changelog update.
  - Clear README including:
    - How to configure secrets (LLM API key).
    - How path and command restrictions work.
    - Security caveats and recommended safe defaults.

NON‑GOALS FOR MVP
- Full multi‑agent orchestration.
- Supporting arbitrary system package installs.
- Complex approval UIs; rely on standard GitHub PR review and artifacts.

QUALITY BAR
- Action must be deterministic and fail closed (if manifest invalid, or attempt to violate policy → job fails with clear error).
- Minimal tests for the policy engine and path enforcement.
- Clean logs; no noisy debug output by default.

DELIVERABLES
- GitHub Action definition (action.yml, Dockerfile or composite).
- Agent runner (Python or Node) implementing sandbox logic and LLM interaction.
- Manifest schema and validation.
- Example workflows and sample repositories to demonstrate usage.
